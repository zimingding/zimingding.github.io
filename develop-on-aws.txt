12:Tab4,<p>Last month I attended a instructor-lead training from AWS called "Developing on AWS". The content is really great and lots of hands-on labs that I have learned a lot.</p>
<h2>Configure Development Environment</h2>
<p>There are many ways to configure development environment: aws configure, aws login, manually enter access key etc. The goal is to be able to provision AWS services using CLI. One thing to remember is behind the scenes AWS console is calling CLI/SDK to do everything. So you don't have to use console to manage AWS resources.</p>
<pre><code>aws --version
aws s3 ls
aws s3 rb s3://$bucketToDelete
</code></pre>
<h2>S3</h2>
<p>Low-level API</p>
<pre><code>s3client = boto3.client('s3')
</code></pre>
<p>High-level API</p>
<pre><code>s3resource = boto3.resource('s3')
</code></pre>
<p>We normally use boto3.client which is low-level API</p>
<ul>
<li>s3Client.create_bucket</li>
<li>s3Client.get_waiter('bucket_exists')</li>
<li>s3Client.upload_file</li>
<li>s3Client.download_fileobj</li>
<li>s3Client.put_object</li>
<li>s3Client.put_bucket_website</li>
</ul>
<p>A presigned URL is perfect for giving temporary, secure access to a S3 object without making the bucket public. Here's how it works:</p>
<ol>
<li>You generate a URL that includes a signature and expiration time.</li>
<li>Anyone with the URL can access the object (read or write, depending on teh method) until it expires.</li>
<li>The bucket and object remain private; no public ACL or bucket policy is needed.</li>
</ol>
<h2>DynamoDB</h2>
<ul>
<li>ddbClient.create_table</li>
<li>ddbClient.query</li>
<li>ddbClient.update_item</li>
</ul>
<p>It makes more sense to create table in AWS console or use database migration as you usually just create it once, it can be done via boto3 client of course.</p>
<p>Global Secondary Index vs Local Secondary Index</p>
<p>Eventual consistency vs Strong consistency, eventual consistency costs 0.5 RCU as it uses less resources but can get stale data.</p>
<h2>Lambda</h2>
<p>Lambda is super useful, it sits behind API Gateway for all the applications we build. It's serverless, automatically scale to up to 1000 instances and can even increase, use layers to reduce duplication, save in /tmp folder to improve performance.</p>
<p>Upload Lambda via zip file or container, zip file is the easier method.</p>
<pre><code>aws lambda update-function-code --function-name dictate-function --zip-file fileb://dictate-function.zip
</code></pre>
<h2>API Gateway</h2>
<p>It creates a http endpoint (e.g. REST) to integrate with Lambda.</p>
<p>To create a route we need to create resources, method, we can use mapping templates to map response or request, we can do model validation, enable CORS, and test it before we deploy API.</p>
